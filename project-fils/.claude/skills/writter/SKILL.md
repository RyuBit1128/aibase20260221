# Writer Skill - 記事作成スキル

ユーザーからテーマ・要件を受け取り、ヒアリング → 構成案 → 執筆 → 品質チェック → 保存の5ステップで記事を完成させる。
ユーザーの承認なしに次のステップへ進まないこと。記事の品質はユーザーとの対話を通じて担保する。

## トリガーワード

以下のいずれかのワードでスキルが起動する：

- `記事を書いて`
- `記事作成`
- `ライティング`
- `ブログを書いて`
- `コンテンツ作成`
- `write article`
- `/write`

## 対応する記事タイプ

- **blog** - 技術ブログ、ナレッジ共有
- **seo** - 検索エンジン最適化を意識したコンテンツ
- **column** - コラム、オピニオン記事
- **howto** - ハウツー、チュートリアル記事
- **news** - ニュース、レポート記事

## 利用フロー

### Step 1: キャラクター読み込み・ヒアリング（要件定義）

まず `character/persona.md` を読み込み、以下を把握した上で全ての工程に反映する：

- 文章のトーン（ゆるく、やわらかく）
- 言葉選びのルール（専門用語の言い換え）
- 共感ファーストの姿勢
- 記事タイプに応じたライティングテンプレート（AIDMA / PREP / SDS）

次に、記事の品質はヒアリングの精度で決まる。AskUserQuestionを使い、以下を確認すること。
ユーザーが曖昧な回答をした場合は具体例を示して再確認する。

| 確認項目 | 必須 | 例 |
|---------|------|-----|
| テーマ・タイトル | 必須 | 「Next.js 15の新機能まとめ」 |
| 記事タイプ | 必須 | blog / seo / column / howto / news |
| ターゲット読者 | 必須 | 初心者エンジニア / 経営者 / 一般読者 |
| 文字数目安 | 任意 | 2000字 / 5000字 |
| トーン | 任意 | カジュアル / フォーマル / 技術的 |
| キーワード | 任意 | SEO対策キーワード |
| 参考情報・URL | 任意 | 参考にすべきソース |

### Step 2: 構成案の作成

ヒアリング結果をもとにアウトラインを作成し、ユーザーに提示する。
構成案は記事全体の骨格であり、ここでの合意が最終品質を左右する。

```
## 構成案
1. 導入（リード文）
2. 見出し1: ○○○
   - サブ見出し1-1
   - サブ見出し1-2
3. 見出し2: ○○○
4. 見出し3: ○○○
5. まとめ
```

**ユーザー承認を待つ** - 構成案がOKになるまで次に進まない。

### Step 3: 本文執筆

承認された構成案に基づき、マークダウン形式で執筆する。

執筆ルール：
- `#` はタイトル専用。本文の見出しは `##` から開始する
- 段落間は空行を入れる
- 箇条書き・表・コードブロックを適切に活用し、壁のような文章を避ける
- 画像の挿入位置には `<!-- IMAGE: 説明 -->` プレースホルダーを配置する

### Step 4: 品質チェック

執筆完了後、以下の観点でセルフチェックを実施する。
品質チェックは記事の信頼性を守るためのゲートであり、基準を満たさない記事は修正してから提出する。

| チェック項目 | 基準 |
|-------------|------|
| 誤字脱字 | 明らかな誤字がないこと |
| 文章の一貫性 | トーン・文体が統一されていること |
| 構成の論理性 | 導入 → 本論 → 結論の流れが自然であること |
| 見出しの適切さ | 見出しが内容を正しく反映していること |
| 読みやすさ | 一文が長すぎないこと（目安: 60文字以内） |
| ターゲット適合 | 読者レベルに合った表現であること |
| メタ情報 | frontmatterが正しく記入されていること |

チェック結果をユーザーに報告し、修正が必要な場合は対応する。

### Step 5: 最終確認・保存

品質チェック通過後、最終版をユーザーに提示する。承認を得てからファイルに保存する。
承認なしにファイルを保存しないこと。

## 出力先ディレクトリ

記事タイプごとにサブディレクトリを分ける。存在しない場合は作成する。

```
project-fils/output/articles/
├── blog/       # ブログ記事
├── seo/        # SEO記事
├── column/     # コラム記事
├── howto/      # ハウツー記事
├── news/       # ニュース記事
└── drafts/     # 下書き・中断時の保存先
```

## ファイル名規則

**形式:** `YYYY-MM-DD-slug.md`

- 日付はファイル作成日（ISO 8601形式）
- スラッグは英数字・ハイフンのみ（小文字）
- スペースはハイフンに変換
- 日本語はローマ字または英語に変換

**例:**
```
2026-02-21-nextjs15-new-features.md
2026-02-21-ai-agent-introduction.md
2026-02-21-typescript-best-practices.md
```

## 記事テンプレート

全記事は以下のマークダウン構成に従う：

```markdown
---
title: "記事タイトル"
date: YYYY-MM-DD
type: blog | seo | column | howto | news
author: ""
tags: []
target: "ターゲット読者"
status: draft | review | published
description: "記事の概要（120文字以内）"
---

# 記事タイトル

リード文（導入・概要）

## 見出し1

本文...

### サブ見出し1-1

本文...

## 見出し2

本文...

## まとめ

まとめの本文...
```

### frontmatterフィールド

| フィールド | 必須 | 説明 |
|-----------|------|------|
| title | 必須 | 記事のタイトル |
| date | 必須 | 作成日（YYYY-MM-DD） |
| type | 必須 | 記事タイプ（blog / seo / column / howto / news） |
| author | 任意 | 著者名 |
| tags | 任意 | 関連タグの配列 |
| target | 必須 | ターゲット読者 |
| status | 必須 | draft / review / published |
| description | 必須 | 記事の概要（SEO用、120文字以内） |

## 注意事項

- **ユーザー承認なしに記事を保存しない** - 構成案と最終版で必ず確認を取る
- **既存ファイルを上書きしない** - 同名ファイルが存在する場合はサフィックス（`-v2`等）を付与する
- **著作権に配慮する** - 他サイトのコピーや過度な引用を行わない
- **事実確認を意識する** - 技術的な内容は正確性を重視し、不確実な情報には注記を付ける
- **フロントマター必須** - 全記事にメタ情報を含むfrontmatterを付与する
- **下書き保存** - 途中で中断する場合は `drafts/` ディレクトリに保存する

## スキルのファイル構成

このスキル自体のファイル構成。記事執筆時に参照するリソースをここに配置する。

```
.claude/skills/writter/
├── SKILL.md                # スキル定義ファイル（本ファイル）
└── character/
    └── persona.md          # ライターのペルソナ・キャラクター設定
```

| ファイル | 役割 |
|---------|------|
| `SKILL.md` | スキルの定義・フロー・ルールを記述する本体ファイル |
| `character/persona.md` | 記事を書く際のトーンや人格設定。記事タイプやターゲットに応じて参照する |
